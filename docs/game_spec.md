# 🚀 Project We-Go: Game Specification Document

## 1. 게임 콘셉트 개요
은하영웅전설 스타일의 우주 함대 전투를 모티브로 한 **동시 턴제(We-Go Turn-based)** 전술 게임입니다.

### 핵심 메커니즘
* **관성 이동 (Vector Movement):** 우주선은 관성의 영향을 받으며, 이전 턴의 이동 속도와 방향이 유지됩니다. 여기에 새로운 추진력을 더해 최종 목적지가 결정됩니다.
* **선회각 제한:** 거대한 전함의 특성을 반영하여 한 턴에 회전할 수 있는 각도(부채꼴)에 제한이 있습니다.
* **방향성 장갑:** 정면은 방어력이 높지만 측면이나 후방은 취약하여, 상대의 경로를 예측해 배후를 노리는 것이 핵심 전략입니다.
* **무기 체계 (예측 사격):**
  * **빔 병기:** 직사 무기로, 발사 즉시 도달하며 궤적 상의 적을 타격합니다.
  * **유도 미사일 / 범위 폭격:** 목표 좌표에 도달하거나 특정 영역을 타격하는 무기로, 상대의 다음 위치를 예측해야 합니다.
* **동시 처리 시뮬레이션 ("펑펑펑" 단계):** 모든 플레이어의 명령이 모이면 타임라인(예: 5초)에 따라 이동, 사격, 피격이 동시에 연출되는 시각적 카타르시스를 제공합니다.

## 2. 개발 로드맵
프로젝트는 1:1 프로토타입에서 대규모 함대전으로 점진적으로 확장합니다.

* **Phase 1: 코어 메카닉 검증 (1:1 단일 함선 대전)**
  * "관성 이동 + 예측 사격" 코어 루프 구현
  * 2D 벡터 물리(관성, 선회 제한) 및 기본 무기 메커니즘 테스트
* **Phase 2: 심리전 고도화 (1:1 전술 다각화)**
  * 전장의 안개(Fog of War) 도입 및 함선 클래스 추가
  * 에너지 분배 시스템(추진/방어/무기)으로 하이 리스크-하이 리턴 심리전 강화
* **Phase 3: 소대급 확장 (多 대 多 매치)**
  * 3~5척의 함선을 조작하는 진형(Formation) 및 협동 무기 체계 도입
  * 동시 명령을 위한 직관적인 UI/UX 고도화
* **Phase 4: 은하영웅전설의 완성 (대규모 함대 전술)**
  * 기함과 다수의 군집(Flock Physics) 조작
  * 진형 붕괴, 사기(Morale), 시네마틱 스케일 연출 추가

## 3. 기술 스택 및 아키텍처
비용 절감과 확장을 위해 **'결과값 데이터화(Deterministic Simulation)'** 아키텍처를 채택합니다. 서버는 그래픽이나 실시간 루프를 돌리지 않고, 오직 **수학적 물리 계산**만 수행하여 JSON 데이터로 응답합니다.

### 추천 기술 스택
* **클라이언트 (Frontend):** Unity (C#) - WebGL 빌드 타겟 추천
  * *패키지:* Newtonsoft JSON (데이터 직렬화), DOTween (애니메이션/연출), UnityWebRequest (통신)
* **서버 (Backend):** Node.js (TypeScript) + Express
  * *특징:* 무상태(Stateless) API 또는 가벼운 인메모리 처리로 설계하여 향후 서버리스(AWS Lambda) 전환 용이. 물리 수식 오류 방지를 위해 TypeScript 사용.
  * *패키지:* Express, dotenv

### 데이터 흐름 구조
```
[플레이어 A/B 클라이언트 (Unity)] 
       │ 
       │ (1) 명령 제출 (이동 벡터, 무기 좌표) - HTTP POST / JSON
       ▼
[게임 서버 (Node.js + Express)]
       │
       │ (2) 두 유저의 데이터가 모두 모였는지 확인 (In-Memory Room State)
       │ (3) 0초~5초까지의 타임라인 물리 시뮬레이션 계산 (관성, 충돌 판정)
       ▼
[JSON 결과 데이터 생성]
       │
       │ (4) 계산 결과 응답 (시뮬레이션 타임라인 데이터)
       ▼
[플레이어 A/B 클라이언트 (Unity)]
       │
       └─► (5) 수신한 JSON 데이터를 기반으로 5초간 화려한 연출 재생 ("펑펑펑")
```

## 4. Phase 1 (MVP) 요구사항 및 설계 (PRD)

### 유저 스토리 (Core Loop)
1. **매칭:** 플레이어는 게임에 접속하여 1:1 대전 방에 입장한다.
2. **명령 입력 Phase:** 두 플레이어는 동시에 자신의 함선의 [이동 벡터(추진력/선회각)]와 [발사 무기 좌표]를 지정한다.
3. **명령 제출:** 결정이 끝나면 '완료' 버튼을 누른다. 상대가 완료할 때까지 대기하며, 수정할 수 있다.
4. **시뮬레이션 Phase ("펑펑펑"):** 양쪽 모두 완료하면, 서버가 결과를 계산하여 내려주고, 클라이언트는 5초간의 연출(이동 -> 사격 -> 피격 -> 턴 종료)을 동일하게 재생한다.
5. **승패 판단:** 한 함선의 체력(HP)이 0이 될 때까지 이 루프를 반복한다.

### 기능적 요구사항 (MVP 범위)
* **FR-01 (관성 물리 이동):** 함선은 직전 턴의 속도와 방향(관성)을 유지하며, 이번 턴의 추진력을 더해 최종 궤적을 계산해야 한다.
* **FR-02 (선회력 제한):** 함선은 한 턴에 회전할 수 있는 최대 각도(예: 60도) 제한이 있어야 한다.
* **FR-03 (무기 체계 2종):**
  * *직사 빔:* 발사 궤적 선상에 적이 걸치면 즉시 타격
  * *범위 어뢰:* 특정 좌표 반경 내에 적이 도달하면 폭발 타격
* **FR-04 (동시 턴 처리):** 서버는 두 유저의 명령을 모두 수신한 후에만 물리 계산을 수행하고 결과를 반환해야 한다.

### 세부 설계 (Core Logic)

#### 1. 서버 사이드: 물리 및 턴 처리 로직 (Node.js)
서버는 시뮬레이션 틱(예: 1초에 10번 계산, 5초 동안 총 50틱)을 돌리며 좌표를 추적합니다.

* **함선 상태 데이터 구조 (State):**
```typescript
interface ShipState {
  position: { x: number, y: number }; // 현재 좌표
  velocity: { x: number, y: number }; // 현재 속도 벡터
  heading: number;                   // 현재 바라보는 각도 (라디안)
}
```

* **물리 계산 공식 (Vector Math):**
  * 새로운 속도 $V_{new} = V_{old} + \text{추진력 벡터}$
  * 새로운 위치 $P_{new} = P_{old} + V_{new}$
  * 선회각 제한 체크: $\lvert\text{목표 각도} - \text{현재 각도}\rvert \le \text{최대 선회각}$ 인지 검증하여 치팅 방지.

#### 2. 클라이언트 사이드: UI/UX 및 연출 (Unity)
유저에게 스트레스를 주지 않는 친절한 UI 가이드가 심리전의 핵심입니다.

* **예측 가이드선 UI:** 플레이어가 추진력 게이지를 올리거나 조타 핸들을 돌릴 때, "아무런 외력이 없을 때 미끄러지는 예상 잔상(Ghost)"과 "조작이 반영된 최종 예상 위치"를 실시간으로 화면에 선(LineRenderer)으로 그려줍니다.
* **결과 연출 시퀀서 (Sequence Parser):** 서버에서 응답받은 데이터가 `[{time: 1.2, event: "MOVE", x: 12, y: 15}, {time: 3.0, event: "FIRE", type: "BEAM"}, {time: 3.1, event: "HIT", damage: 20}]` 와 같은 형태라면, 클라이언트는 `DOTween`을 이용해 해당 타임코드에 맞춰 이벤트를 순차 실행합니다.
